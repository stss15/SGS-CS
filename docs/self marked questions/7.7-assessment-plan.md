## Question Design

### Q1: Trace Table Purpose (Multiple Choice)
**Concept**: Why use trace tables?
-   **Question**: What is the primary purpose of a trace table?
-   **Options**:
    -   To record the value of variables at each step (Correct)
    -   To write the code for the program
    -   To speed up the computer
    -   To design the user interface

### Q2: Dry Run Logic (Ordering)
**Concept**: Steps in a dry run.
-   **Order**:
    1.  Set up columns for each variable and output
    2.  Select a set of test data
    3.  Work through the algorithm step-by-step
    4.  Record changes in variable values in the table
    5.  Compare actual output with expected output

### Q3: Variable Tracking (Drag & Drop)
**Concept**: Tracking a simple loop.
-   **Snippet**:
    ```
    Count <- 0
    FOR X <- 1 TO 3
       Count <- Count + X
    NEXT X
    ```
-   **Buckets**: Value of Count (End of Loop 1), Value of Count (End of Loop 2), Value of Count (End of Loop 3)
-   **Items**:
    -   1 (Loop 1)
    -   3 (Loop 2)
    -   6 (Loop 3)
    -   0 (Initial)
    -   2 (Distractor)
    -   4 (Distractor)

### Q4: Output Prediction (Multiple Choice)
**Concept**: Predicting output from a trace.
-   **Snippet**:
    ```
    A <- 5
    B <- 10
    IF A < B THEN
       OUTPUT "Less"
    ELSE
       OUTPUT "More"
    ENDIF
    ```
-   **Question**: What is the output?
-   **Options**:
    -   Less (Correct)
    -   More
    -   5
    -   10

### Q5: Algorithm Analysis (Matching)
**Concept**: Linking trace table terms.
-   **Pairs**:
    -   Dry Run -> Manual execution of an algorithm
    -   Variable Column -> Records value changes
    -   Output Column -> Records what is displayed
    -   Test Data -> Input values used for the trace
