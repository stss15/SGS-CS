The Parity Bit

How can one extra bit help us detect errors?

Simple Trick: Add an extra bit to make the total count of 1s either even or odd.

If it doesn't match when received, we know there's an error!

Even Parity

Total number of 1s must be Even

Example: If 3 ones, add a 1 (Total 4 - even!)

If 4 ones, add a 0 (Total 4 - already even!)

Like checking if a number is divisible by 2.

Odd Parity

Total number of 1s must be Odd

Example: If 3 ones, add a 0 (Total 3 - odd!)

If 4 ones, add a 1 (Total 5 - odd!)

Opposite of even - must end in odd count.

The Parity Block

What if two bits flip? Simple parity fails!

Simple Parity Problem

The Flaw: If 2 bits flip, the check passes!

Example: 3 ones → 2 bits flip → 1 one (still odd!)

Error not detected.

Like two wrongs making a right - the count still matches!

Parity Block Solution

Arranging data in a grid

Checking rows AND columns

Benefit: Can identify exactly which bit flipped

By cross-referencing row and column, we can correct it!

Like coordinates on a map - row and column pinpoint the error.

The Checksum

How can we use math to verify data integrity?

Analogy: Like a receipt total. If the numbers don't add up, something's wrong!

1. Calculate

Sender calculates a value from the data (using math).

Like adding up all the numbers on a receipt.

2. Send

Sends data + checksum value.

Like sending the receipt with the total.

3. Verify

Receiver recalculates the checksum.

If values match OK

If not Error

Like checking if your calculation matches the receipt total.

Real-World Use: Credit Cards, ISBNs, File downloads (to verify file integrity)

Other Methods

What if we just ask "did you get that?"

Echo Check

Receiver sends the data back to the sender

Sender compares original with echo

Problem: Inefficient (doubles the data sent)

Like repeating everything you say - wastes bandwidth.

ARQ

Automatic Repeat Request

"Did you get that?"

More efficient - just sends a confirmation, not the whole message

Like a simple "yes" or "no" response.

ARQ (Automatic Repeat Request)

How does ARQ automatically handle errors?

1. Acknowledgement (ACK)

Receiver confirms receipt

Sends a small "ACK" message back

Like saying "got it!" after receiving a message.

2. Timeout

Sender waits for ACK

If no ACK received in time, Resend

Like waiting for a reply - if no answer, ask again!

Automatic: The system handles this without human intervention.

Misconception Check

  1. "A parity check can detect if 2 bits flip."
  2. "A parity block check can correct errors."
  3. "ARQ relies on the receiver sending the data back."

Press Down to reveal answers

Question 1

"A parity check can detect if 2 bits flip."

Answer: False

It will still look even/odd (e.g. two 0s become two 1s, count increases by 2, parity remains same).

Question 2

"A parity block check can correct errors."

Answer: True

By cross-referencing row and column, the exact bad bit is found.

Question 3

"ARQ relies on the receiver sending the data back."

Answer: False

That's Echo Check. ARQ sends an Acknowledgement (ACK).
