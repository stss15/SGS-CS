7.1 Program Development Life Cycle

Topic 7: Algorithm Design and Problem Solving

The Big Question

Would you build a house without a blueprint?

Imagine a builder just turning up with some bricks and saying "I'll figure it out as I go..."

The same is true for software.

Without a plan, programs become chaotic, full of bugs, and impossible to finish.

The Solution

The Program Development Life Cycle

Every program goes through the same journey:

1. Analysis

What is needed?

2. Design

How will it work?

3. Coding

Build it!

4. Testing

Does it work?

We'll explore each stage in detail...

Stage 1: Analysis

What problem are we actually solving?

Before writing any code, we must deeply understand:

  • What does the user actually need?
  • What data goes in?
  • What information comes out?
  • What rules must be followed?

Output: A Requirements Specification – a clear document of exactly what the program must do.

Two Powerful Tools

Analysis Tools: Abstraction

How do we focus on what matters?

Think of a Tube Map

The real London Underground has curved tunnels, exact distances, and messy details.

But the famous Tube Map ignores all of that. It shows only what matters: stations and connections.

Abstraction

Definition: Removing unnecessary details to focus on the essential elements needed to solve the problem.

Keep what you need. Discard what you don't.

Analysis Tools: Decomposition

How do we tackle a massive problem?

Example: "Make breakfast" seems simple. But it's actually:

       MAKE BREAKFAST
            │
    ┌───────┼───────┐
    │       │       │
  Toast   Eggs    Tea
    │       │       │
 ┌──┴──┐ ┌──┴──┐ ┌──┴──┐
Get   Put Crack Boil Fill
Bread in  Eggs  Water Kettle
      Toaster

Decomposition: Breaking a complex problem into smaller, manageable sub-problems that are easier to solve.

Stage 2: Design

How will the solution actually work?

Now we plan how to solve the problem:

  • What steps does the program follow?
  • What decisions does it make?
  • How is data stored and processed?

We use special tools to document our design...

Design Tools

Design Tools

Structure Diagrams

Show the hierarchy of a system.

What are the parts and how do they relate?

Flowcharts

Show the steps and their order.

Visual "map" of the logic.

Pseudocode

Describe the logic in plain English-like instructions.

Not real code, but structured like it.

We'll learn all three in detail in the coming lessons!

Stage 3: Coding

Now we write the actual program!

Using the design documents, the programmer writes code in a programming language.

Modular Development

Each sub-system from the design is coded as a separate module.

Modules can be built and tested independently.

Iterative Testing

As each module is written, it is tested immediately.

Fix bugs as you go, not at the end!

Stage 4: Testing

Does it actually work correctly?

The complete program is tested with many different sets of data to ensure:

  • It produces the correct outputs
  • It handles unexpected inputs gracefully
  • All parts work together as a whole

Key Idea: Testing is not just "trying it out". It's a systematic process using carefully chosen test data.

We'll cover this in detail in lesson 7.7!

The Complete Cycle

Why is it called a "cycle"?

Because development rarely goes in a straight line!

    ┌─────────────────────────────────────┐
    │                                     │
    ▼                                     │
 ANALYSIS ──► DESIGN ──► CODING ──► TESTING
                │                    │
                └────────────────────┘
                   (Go back and fix!)

Testing might reveal design problems. Design might need to revisit analysis. It's iterative!

Real World Example

Building a School Registration System

Walk through each stage

Stage 1: Analysis

Question: What does a registration system need to do?

Inputs

  • Student names
  • Present/Absent status
  • Date

Processes

  • Record attendance
  • Calculate % attendance
  • Flag poor attendance

Outputs

  • Daily register
  • Reports for parents
  • Alerts for teachers

Stage 2: Design

Structure Diagram:

       REGISTRATION SYSTEM
              │
    ┌─────────┼─────────┐
    │         │         │
Take     Generate    Send
Register  Reports   Alerts

Then we'd write pseudocode or flowcharts for each sub-system...

Stages 3 & 4: Coding & Testing

Coding

Each module (Take Register, Generate Reports, Send Alerts) is programmed separately.

Testing

Test with:

  • Normal data (regular students)
  • Edge cases (100% attendance, 0%)
  • Invalid data (blank names)

Key Takeaways

  1. The PDLC has 4 stages: Analysis, Design, Coding, Testing
  2. Abstraction = Remove unnecessary detail
  3. Decomposition = Break down complex problems
  4. Design uses Structure Diagrams, Flowcharts, and Pseudocode
  5. Development is iterative – you go back and fix!

True or False?

Check Your Understanding

  1. Coding should be the first step when creating a program.
  2. Abstraction means adding more detail to a problem.
  3. The output of Analysis is a requirements specification.

Press Down to reveal answers

Question 1

Coding should be the first step when creating a program.

Answer: FALSE

Analysis comes first! You must understand the problem before you can solve it.

Question 2

Abstraction means adding more detail to a problem.

Answer: FALSE

Abstraction is the opposite – it means removing unnecessary detail to focus on what matters.

Question 3

The output of Analysis is a requirements specification.

Answer: TRUE

The requirements specification documents exactly what the program must do, based on the analysis of the problem.
